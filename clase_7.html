! EL PROYECTO: TU PROPIO CHATBOT

En las clases 4, 5 y 6 aprendimos a conectarnos a los servidores de Groq, a armar preguntas dinámicas usando variables y a crear escudos (try/except) para atrapar errores. Hoy llegó el momento de integrar todo eso en un proyecto real.

El objetivo de este Trabajo Práctico es que construyan desde cero un Chatbot interactivo completo. No van a hacer un programa que responda una sola pregunta y se cierre; van a programar un sistema continuo donde el usuario pueda configurar un personaje y chatear fluidamente por la consola.

Ustedes van a ser los arquitectos completos: van a armar el "backend" (la función que se conecta al cerebro de la IA), el "frontend" (la interfaz que le pide los datos al usuario) y finalmente un Informe de Control de Calidad (QA) para documentar cómo probaron su sistema.

? REGLAS DEL JUEGO Y MODALIDAD

REGLA DE ORO (COMPRENSIÓN > MEMORIA): No queremos que memoricen sintaxis. Pueden y deben revisar sus cuadernos anteriores para extraer código. Lo que se evalúa es que sepan ensamblarlo. Por esto, es OBLIGATORIO comentar (#) cada línea o bloque clave explicando qué hace. Quien no puede explicar su código, no aprueba. Si vas a usar la IA, tené en cuenta que el código lo tenés que explicar.
👥 MODALIDAD Y GRADOS DE DIFICULTAD: El TP puede hacerse individual o en grupos de hasta 3 personas. La exigencia técnica sube según la cantidad de integrantes:
  • Individual: Bucle Continuo + Informe QA. El chatbot debe estar dentro de un bucle while True para pedir personajes y temas infinitamente hasta que el usuario escriba "salir".
  • Parejas (2): Bucle + Memoria + Comandos Ocultos. Además del bucle continuo, deben implementar una lista historial = []. Deben agregar comandos especiales de sistema: si el usuario escribe /historial, el programa imprime la memoria; si escribe /limpiar, la memoria se borra.
  • Tríos (3): Memoria + Comandos + Guardrails (Seguridad). Además de lo anterior, deben programar un filtro lógico (un if) antes de la conexión. Si el texto del usuario contiene palabras de una "lista negra", el programa debe rechazar la consulta sin gastar tokens.

  • Dentro de la función, armar el payload inyectando los parámetros. Usar f-strings para armar el prompt dinámico.

💻 FASE 2: FRONTEND E INFORME DE TESTING (Lunes - 120 min)

2 INTERFAZ Y DOCUMENTACIÓN

Con el motor funcionando, hoy armamos la interacción con el usuario y el reporte de calidad.

Módulo 2: Interfaz de Consola (Frontend)

Nada puede estar "hardcodeado". Todo se pide en tiempo de ejecución:

  • Implementar un bucle while True para que el programa corra continuamente.
  • Usar input() para pedir el Personaje y el Tema. (Manejar el comando "salir" para romper el bucle).
  • Usar input() para pedir la Temperature (usando float() para convertir el texto a decimal).
  • Llamar a la función, imprimir el resultado, e implementar el Historial, Comandos Ocultos o Guardrails según la cantidad de integrantes de su grupo.

Módulo 3: El Informe de QA (Control de Calidad)

Debajo de su código, agreguen una Celda de Texto en Colab. Van a redactar un informe profesional que contenga:

  1. Ficha Técnica: ¿Qué modelo exacto de Groq utilizaron? ¿Qué valor de temperature consideran ideal para este chatbot y por qué?
  2. Prueba de Red (Crash Test): Borren una letra de su API_KEY a propósito y ejecuten el programa. Copien y peguen en el informe qué respondió su bloque try/except para demostrar que el programa no colapsó con letras rojas.
  3. Prueba de Alucinación: Documenten una pregunta local o imposible que le hayan hecho a su chatbot donde la IA haya inventado datos.
  4. Prueba de Seguridad (Solo grupos de 3): Muestren cómo reaccionó el sistema al ingresar una palabra de la lista negra.
